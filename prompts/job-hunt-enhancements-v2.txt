### Step 0 — JD requirements extractor (pure Python, no API call)

Write this function in fit_analyzer.py. It extracts only the requirements/
qualifications section from a JD before sending to Groq — no boilerplate,
no "about us", no benefits copy.

```python
# Section-start triggers — any line containing one of these (case-insensitive)
# signals the start of the requirements block
REQUIREMENTS_TRIGGERS = [
    "requirement", "qualification", "what you'll need", "what we're looking for",
    "what you bring", "you have", "you will have", "you should have",
    "you must have", "must have", "must-have", "basic qualifications",
    "minimum qualifications", "preferred qualifications", "nice to have",
    "nice-to-have", "skills required", "skills we're looking for",
    "experience required", "experience we're looking for",
    "about you", "who you are", "what you need", "technical requirements",
    "technical skills", "key skills", "core skills", "competencies",
    "the ideal candidate", "we are looking for", "we're looking for",
    "successful candidate", "you will need",
]

# Section-end triggers — stop collecting when one of these appears
# after the requirements section has started
STOP_TRIGGERS = [
    "what we offer", "what we provide", "we offer", "benefits",
    "perks", "compensation", "salary", "pay range", "total rewards",
    "about us", "about the company", "about {", "our culture",
    "why join", "why work", "equal opportunity", "diversity",
    "accommodation", "accessibility", "application process",
    "how to apply", "next steps",
]

def extract_requirements(jd: str) -> str:
    """
    Extract the requirements/qualifications section from a job description.
    Returns up to 1500 chars of the requirements block.
    Falls back to the first 1500 chars of the full JD if no trigger is found.
    Never raises — always returns a non-empty string.
    """
    lines = jd.splitlines()

    # Find where requirements section starts
    start = None
    for i, line in enumerate(lines):
        stripped = line.strip().lower()
        if any(t in stripped for t in REQUIREMENTS_TRIGGERS):
            start = i
            break

    # No trigger found — fall back to first 1500 chars of full JD
    if start is None:
        return jd[:1500]

    # Find where requirements section ends
    end = len(lines)
    for i, line in enumerate(lines[start + 1:], start + 1):
        stripped = line.strip().lower()
        if any(t in stripped for t in STOP_TRIGGERS):
            end = i
            break

    section = "\n".join(lines[start:end]).strip()

    # If extraction yielded almost nothing (header only), fall back
    if len(section) < 80:
        return jd[:1500]

    return section[:1500]
```

### Step 1 — Seniority detection (pure Python, no API call)
Used only to LABEL the job and the user for context in the Groq prompt.
NOT used to filter or drop any job.

```python
def detect_seniority(job_title: str, job_description: str) -> str:
    """Returns: "junior" | "mid" | "senior" | "lead" | "any" """
    text = (job_title + " " + job_description[:300]).lower()
    if any(w in text for w in ["principal", "staff", "vp ", "director", "head of", "architect"]):
        return "lead"
    if any(w in text for w in ["senior", "sr.", "sr ", "lead ", "8+ years", "7+ years", "6+ years"]):
        return "senior"
    if any(w in text for w in ["junior", "jr.", "jr ", "entry", "entry-level", "new grad",
                                "graduate", "intern", "0-2 years", "1-2 years", "1+ year"]):
        return "junior"
    if any(w in text for w in ["mid ", "mid-level", "intermediate", "3-5 years", "2-4 years"]):
        return "mid"
    return "any"


def detect_user_level(resume_text: str) -> str:
    """Returns: "student" | "junior" | "mid" | "senior" """
    text = resume_text.lower()
    years_exp = _estimate_years(text)
    if any(w in text for w in ["currently enrolled", "bachelor", "bsc", "b.sc", "honours",
                                "expected graduation", "gpa", "university"]):
        if years_exp < 2:
            return "student"
    if years_exp < 2:
        return "junior"
    if years_exp < 5:
        return "mid"
    return "senior"


def _estimate_years(text: str) -> int:
    import re
    years = re.findall(r'20\d{2}', text)
    return len(set(years)) // 2 if years else 0
```

These values are passed as context to Groq so it can grade more accurately.
They do NOT gate any job from being graded.

- Model: llama-3.3-70b-versatile
- Max tokens: 220
- Temperature: 0.3
- System prompt (exact):

   SUMMARY: [one honest sentence]
   STRENGTHS: [strength 1] | [strength 2] | [strength 3]
   GAPS: [gap 1] | [gap 2] | [gap 3]

  ← Do NOT pass the full JD — extract_requirements() handles the selection.

### Returns this dict (always — never None):
```python
{
    "grade":      str,        # "A+" / "B" / "C-" / "D" / "F" etc.
    "seniority":  str,        # detected job seniority label
    "user_level": str,        # detected user level from resume
    "summary":    str,        # 1 sentence honest fit assessment
    "strengths":  list[str],  # max 3
    "gaps":       list[str],  # max 3
    "tip":        str,        # 1 sentence cover letter tip

```

Safe defaults if Groq fails or response is malformed:
  grade="?", summary="Could not evaluate", strengths=[], gaps=[], tip="Apply anyway"
  seniority and user_level are always filled from pure-Python detection — never fail.

### Response parsing:
- Split by newlines, find each label prefix (GRADE:, SUMMARY:, etc.)
- Split STRENGTHS/GAPS by " | "
- If any field fails to parse: use safe defaults for that field only, never raise
- Strip whitespace from all values

## TASK — Part B: Portal scanner service

Create backend/app/services/portal_scanner.py

### ATS-specific scrapers (use httpx, NOT Playwright where possible):

**Greenhouse** (API-first — no Playwright needed):
```
GET https://boards-api.greenhouse.io/v1/boards/{greenhouse_id}/jobs?content=true
```
Returns JSON. Parse `jobs[]` array. Each job has: `title`, `absolute_url`, `location.name`, `content` (HTML description).
Filter by keyword if provided. Return list of normalized job dicts.

**Lever** (API-first — no Playwright needed):
```
GET https://api.lever.co/v0/postings/{lever_id}?mode=json
```
Returns JSON array. Each posting has: `text` (title), `hostedUrl`, `categories.location`, `descriptionPlain`.
Filter by keyword if provided.

**Ashby** (API-first — no Playwright needed):
```
POST https://api.ashbyhq.com/posting-api/job-board/{ashby_id}
Body: {"jobBoardName": "{ashby_id}"}
```
Returns JSON with `jobPostings[]`. Each has: `title`, `jobUrl`, `teamName`, `locationName`, `descriptionHtml`.

**Custom ATS (Playwright fallback):**
- Use existing Playwright setup from the scrapers/ directory
- Navigate to URL, wait for job listings to load
- Extract: job title, apply URL, location text
- Timeout: 15s per portal — skip and log if exceeded
- Only use Playwright for custom ATS portals, never for Greenhouse/Lever/Ashby

### Main function:
```python
async def scan_canadian_portals(
    keywords: list[str],
    provinces: list[str] | None = None,   # filter by province codes e.g. ["ON", "BC"]
    ats_types: list[str] | None = None,    # filter to only scan certain ATS types
    max_results_per_portal: int = 10
) -> list[dict]
```

- Runs all Greenhouse + Lever + Ashby portals concurrently (httpx async)
- Runs custom ATS portals sequentially with Playwright (too risky to parallelize)
- Deduplicates results by URL
- Normalizes all results to the same shape as jobs from the existing scrapers
  (check the existing Job model fields and match them exactly)
- Returns combined list sorted by portal name

### New API endpoint in jobs.py:
```
POST /api/jobs/scan-canada
Body: {
  "keywords": ["cloud engineer", "devops"],
  "provinces": ["ON", "BC"],   // optional
  "session_id": "abc123"       // store results in existing session
}
```
- Calls scan_canadian_portals()
- Stores results in the same jobs table as regular search results
- Returns { "found": N, "portals_scanned": N, "session_id": "..." }

### Important constraints:
- Greenhouse/Lever/Ashby calls: use httpx.AsyncClient with 10s timeout
- Custom portals: skip gracefully if they fail — log the error, continue
- Add a User-Agent header on all requests: "Mozilla/5.0 (compatible; JobHuntBot/1.0)"
- Do NOT rate limit the API-based portals (Greenhouse/Lever/Ashby are meant to be called)
- DO add 1s delay between custom ATS Playwright scrapes
